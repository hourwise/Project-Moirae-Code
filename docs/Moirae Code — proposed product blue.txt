Moirae Code — proposed product blueprint

Moirae Code should be a governed, model-agnostic coding environment built from VSCodium, with Ananke, Mnemosyne and Horae running as independent local services.

It is not merely “VSCodium bundled with three MCP servers.” Its defining feature is that every model interaction, tool call, memory operation and consequential action passes through a visible, enforceable runtime contract.

The product promise is:

Use almost any coding model, local or hosted, without surrendering control of what it remembers, what it can access, or what it is allowed to do.

1. The overall stack
Desktop shell
Layer	Proposed technology	Purpose
Editor foundation	VSCodium / Code-OSS	Editing, terminal, Git UI, debugging, extension host
Desktop runtime	Electron, inherited from Code-OSS	Cross-platform desktop application
Moirae integration	First-party VS Code extension	Sidebar, chat, approvals, memory and audit UI
Local control plane	Node.js 22+ / TypeScript	Starts and supervises the Fates
Internal communication	JSON-RPC/MCP over stdio or local IPC	Local, authenticated component communication
Persistent state	SQLite WAL	Policies, audit, model registry, approvals, workspace metadata
Memory storage	Local files plus SQLite indexes	Mnemosyne Compendium
Secrets	OS keychain	API keys and OAuth refresh tokens
Packaging	Platform installers	Windows first, then Linux and macOS

VSCodium is a suitable foundation because it builds open-source binaries from Microsoft’s Code-OSS source and uses Open VSX rather than Microsoft’s restricted extension marketplace. Open VSX can also be self-hosted later for a curated Moirae extension registry.

The Fates
Ananke — authority and outcome control

Ananke answers:

May this action happen?
Does it require approval?
Is the approval still valid?
What exactly happened?
Was the result completed, denied, partial, stale or failed?
What evidence exists?

Ananke owns:

policy decisions;
risk classification;
approval binding;
outcome envelopes;
immutable audit events;
tool-call constraints;
rollback and compensation metadata.
Mnemosyne — controlled project memory

Mnemosyne answers:

What should the model know?
Where did that knowledge come from?
How reliable and current is it?
Does it contradict something else?
May it be included in this context?
What should be forgotten, decayed or revalidated?

Mnemosyne owns:

project onboarding;
memory extraction;
provenance;
confidence and reliability scoring;
contradiction detection;
memory ageing and decay;
context-pack construction;
secrets and sensitive-content exclusion;
project-local memory boundaries.
Horae — orchestration and model mediation

Horae answers:

Which model should handle this request?
Which tools and memories may it receive?
What execution plan should be followed?
Does a local or cloud model satisfy the task and policy?
How are retries, fallbacks and context limits handled?

Horae owns:

model-provider adapters;
chat-session orchestration;
prompt and context assembly;
model capability discovery;
token budgets;
fallback routing;
tool-loop execution;
MCP server discovery;
workflow state;
interruption, resumption and cancellation.

Horae must never bypass Ananke. It proposes and coordinates; Ananke authorises.

Critical boundary

The model must never directly possess unrestricted filesystem, shell, Git, network or credential access.

The required path is:

User request
  → Horae creates a proposed operation
  → Ananke evaluates authority and current state
  → approval if required
  → tool broker executes the narrowly scoped action
  → Ananke records the outcome
  → Mnemosyne optionally records an eligible memory
  → UI presents the result and evidence

This is the heart of Moirae Code.

3. The Laws of Moirae Code

These should be normative, testable product laws rather than marketing principles.

Law I — The user is the final authority

No model, provider, plugin, MCP server or workflow may elevate its own authority.

Administrative policy may restrict what the user can authorise in managed environments, but no component may silently grant itself more power.

Law II — Intent is not authority

A user asking a model to “fix everything” is not equivalent to permission to:

delete files;
publish code;
rotate credentials;
deploy;
spend money;
alter permissions;
send external communications;
modify production data.

Intent must be converted into explicit, scoped proposed operations.

Law III — No consequential action without an outcome

Every tool call must end in a typed result such as:

COMPLETED
FAILED
DENIED
WAITING_FOR_APPROVAL
APPROVAL_INVALIDATED
STALE_STATE
TIMED_OUT
PARTIAL_SUCCESS
CANCELLED
COMPENSATION_REQUIRED

Plain text such as “done” is not sufficient evidence.

Law IV — Approval is bound to the exact action

Approval must bind to:

tool identity;
arguments;
affected resources;
content or diff;
repository and branch;
environment;
requesting model;
policy version;
relevant state hash;
expiration time.

Any material mutation invalidates approval.

Law V — Memory is evidence, not truth

Every retained memory must have:

source;
project scope;
timestamp;
provenance;
confidence;
reliability;
sensitivity;
expiry or review status.

A model inference cannot silently become a permanent project fact.

Law VI — Contradictions must remain visible

Mnemosyne must not overwrite inconvenient or conflicting information.

It should preserve the conflict and identify:

both propositions;
their sources;
relative freshness;
confidence;
any chosen working assumption.
Law VII — Least context

A model receives only the minimum memory, files, tools and credentials required for the current task.

A repository-wide request does not automatically justify access to the user’s other repositories, home directory, browser sessions or cloud accounts.

Law VIII — Least capable model that safely completes the task

Horae should route according to:

policy and privacy;
task capability;
context requirements;
reliability;
latency;
cost.

Cost must never override a security restriction or a user’s local-only requirement.

Law IX — Local means local

Under local-only mode:

no prompt leaves the device;
no telemetry leaves the device;
no cloud embeddings are generated;
no remote extension or model is silently used;
no fallback to a hosted provider occurs.

Failure must remain a visible failure, not trigger covert cloud escalation.

Law X — Tools are untrusted until declared and evaluated

A tool description is not proof of behaviour.

Every tool requires a manifest containing:

identity and version;
publisher;
executable or endpoint;
declared permissions;
input and output schemas;
side-effect classification;
network destinations;
data handling;
integrity hash;
signature status.

Unknown capabilities default to denial or isolation.

Law XI — External content cannot grant authority

Repository files, webpages, issues, comments, tool responses and generated text may contain instructions, but none may:

override policy;
approve actions;
expose secrets;
expand scope;
activate new tools.

This directly addresses prompt injection and tool-description injection.

Law XII — Reversibility before autonomy

Where practical, Moirae should prefer:

branch over direct main modification;
draft over send;
preview over publish;
soft delete over permanent delete;
staged diff over blind write;
test environment over production;
reversible migration over destructive migration.
Law XIII — Security decisions happen outside the model

The model may explain risk, but it cannot be the security boundary.

Approval, policy, capability checks, secret access and sandboxing must be deterministic runtime functions.

Law XIV — Every external effect is attributable

The audit record must identify:

initiating user;
active workspace;
originating model/provider;
Horae workflow;
Ananke policy decision;
tool and version;
approval identity;
inputs and hashes;
affected resources;
timestamps;
result and evidence.
Law XV — Failure must be honest

Moirae may not transform:

a partial write into success;
an unverified claim into completion;
a timeout into assumed success;
an unavailable test into a passed test;
a model assertion into execution evidence.
Law XVI — The Fates remain separable

Ananke, Mnemosyne and Horae must each remain usable independently.

Moirae Code composes them; it must not turn them into inseparable editor internals.

This preserves their value as:

standalone MCP infrastructure;
embeddable SDKs;
headless services;
components usable by other IDEs or automation products.

The unique Moirae experience belongs in:

VS Code-compatible extensions can contribute sidebar views, panels, tree views, commands and webview views. Fully custom webviews are available, although official guidance recommends using them only where native views are insufficient.

Current VS Code APIs support chat participants and custom language-model providers, including a one-provider-to-many-models pattern. That is conceptually suitable for Horae’s provider registry. 6. Authentication and identity

Moirae requires several distinct kinds of authentication. They must not be conflated.

A. Local application identity

For a single-user local install:

no mandatory Moirae cloud account;
OS login is the primary local identity;
local workspace profile;
encrypted local key material;
optional application unlock using OS biometrics or password.

The user must be able to use local models, local repositories and all three Fates without creating a PCGsoft account.

B. Model-provider credentials

Store keys in:

Windows Credential Manager;
macOS Keychain;
Linux Secret Service/libsecret.

Never store provider keys in:

settings.json;
repository files;
Mnemosyne memory;
plaintext SQLite;
chat transcripts;
environment dumps;
audit payloads.
C. Git provider authentication

Use established provider mechanisms:

Git Credential Manager;
SSH agent;
device-code OAuth where appropriate;
provider CLI integration such as gh auth.

Moirae should request Git operations through a Git adapter; it should not invent its own storage for GitHub passwords.

D. Remote MCP/API authentication

For HTTP MCP servers, follow the MCP authorisation model and use OAuth-based flows where supported. The MCP specification distinguishes HTTP authorisation from stdio: local stdio servers normally obtain credentials from their environment rather than running the HTTP OAuth flow.

Credentials passed to a local MCP process should be:

server-specific;
minimal scope;
injected only into that child process;
removed when the process exits;
redacted from logs.
E. Optional Moirae account

A PCGsoft/Moirae account should only be required for optional services such as:

licence and subscription management;
encrypted settings synchronisation;
team policy distribution;
managed model gateway;
enterprise audit export;
curated extension registry;
organisation administration.

It must not become a hidden dependency for the local product.

F. Team and enterprise identity

Later editions can add:

OpenID Connect;
SAML through an identity broker;
organisation, team and project roles;
policy bundles signed by administrators;
managed provider configuration;
enterprise MCP authorisation.

MCP’s newer enterprise-managed authorisation work is directly relevant to future team deployments, where individually authorising every server creates operational and security problems.

8. Security architecture
Process separation

Do not run all Fates inside the Moirae extension host.

Recommended processes:

Electron renderer/UI
VS Code extension host
Moirae supervisor
Horae service
Ananke service
Mnemosyne service
one isolated process per external MCP server where practical
sandboxed command/tool runners

This prevents a crash or compromise in one component from automatically exposing everything else.

VS Code already separates extensions through its extension-host architecture, but extensions can still be highly privileged; the official documentation warns that extensions generally have the same permissions as the editor. Therefore the Moirae extension marketplace cannot be treated as intrinsically safe.

Electron requirements

Retain and verify:

renderer sandboxing;
context isolation;
no Node integration in remote content;
restrictive Content Security Policy;
secure-content-only;
explicit permission handlers;
no disabled webSecurity;
no arbitrary navigation from webviews.

These match Electron’s official security recommendations.

Workspace boundaries

Each workspace receives:

unique project identity;
normalised root paths;
allowed path list;
symlink escape protection;
separate memory namespace;
separate audit stream;
separate policy overrides;
explicit multi-root handling.

Block:

../ traversal;
junction or symlink escape;
case-normalisation bypass;
UNC/network path surprises;
alternate data streams where applicable;
writes outside approved roots.
Command sandbox

Classify commands before execution:

Class	Examples	Default
Inspection	git status, typecheck, tests	Allow or low-friction
Workspace mutation	formatter, dependency install	Preview/approval
Host mutation	system package install	Strong approval
Networked mutation	publish, push, deploy	Strong approval
Destructive	delete, database reset	Deny or exceptional approval
Privilege escalation	sudo/admin	Deny by default

Commands should have:

working-directory restriction;
environment allowlist;
timeout;
output cap;
network policy;
filesystem policy;
child-process limit;
resource limits;
cancellation support.
Network control

Introduce an outbound broker with modes:

blocked;
loopback only;
provider endpoints only;
approved domains;
unrestricted with warning.

A model or tool must not create arbitrary outbound connections merely because it can run shell commands.

Secret-loss prevention

Before external model or tool transmission, scan for:

API keys;
private keys;
.env values;
database connection strings;
cloud credentials;
tokens;
personal or regulated data patterns.

The result can be:

redact automatically;
replace with semantic placeholders;
require approval;
block;
force local-model routing.
Extension security

Because extensions are effectively code execution, Moirae needs a curated posture:

signed first-party extensions;
publisher allow/deny policy;
hash verification;
transparent permissions;
version pinning;
vulnerability and reputation data;
quarantine mode;
ability to disable extensions for protected workspaces;
separate “trusted development” and “restricted project” profiles.

VSCodium defaults to Open VSX, and its documentation also permits self-hosted registries. A later Moirae Registry could proxy approved Open VSX packages while publishing first-party extensions, but it should not silently redistribute Microsoft Marketplace packages because of the applicable marketplace restrictions.

Prompt-injection controls

Treat all of these as untrusted data:

source comments;
README files;
issue bodies;
webpages;
MCP tool descriptions;
compiler output;
generated code;
retrieved memory;
model responses.

Controls:

separate instructions from retrieved content;
tag provenance in every context section;
never parse retrieved text as policy;
tool descriptions cannot create permissions;
external content cannot request secrets;
suspicious instructions surfaced to the user;
canary tests for cross-server exfiltration;
taint tracking for sensitive inputs.
Supply-chain security

Required:

locked dependencies;
SBOM for each release;
reproducible-build target;
signed installers;
signed update manifests;
dependency scanning;
secret scanning;
licence scanning;
provenance attestations;
pinned GitHub Actions;
release-channel separation;
rollback mechanism.

14. The most important architectural decision

Do not fork VSCodium deeply to build the intelligence directly into the editor core.

Use:

a thin branded distribution;
a substantial first-party extension;
a separately supervised local control plane;
independent Ananke, Mnemosyne and Horae services.

This gives you:

faster upstream security updates;
less merge debt;
independent testing;
reuse outside Moirae;
replaceable UI;
headless/server operation;
support for other editors later;
a much stronger security boundary.

The Fates should control the coding environment, but they should not be trapped inside it.

Final definition

Moirae Code is best described as:

A local-first, model-agnostic coding environment in which Horae coordinates the work, Mnemosyne governs what is remembered and revealed, and Ananke governs what may actually happen.

Its competitive difference is not another chat box or another code-generating model. It is the enforceable chain between:

request
→ context
→ model
→ proposal
→ authority
→ execution
→ evidence
→ memory

That chain should remain intact regardless of whether the user chooses Claude, OpenAI, DeepSeek, Qwen, a local Llama derivative, or a future model that does not yet exist.
